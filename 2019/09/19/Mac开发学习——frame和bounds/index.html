Mac开发学习——frame和bounds

hhxlz98 · 2019-9-19 · 次阅读

一、前言

对于iOS&Mac开发者来说，frame和bounds肯定不陌生。目前其实有很多讲解frame和bounds的博客，那为什么我还要单独写一篇呢？

  • 博客质量参差不齐，在多浏览几个博客之后反而容易产生疑惑。
  • 初学时仅凭看一下两者的定义就想在实际中自如应用是很难的，使用时可能临场蒙对，但是你不能保证每一次都可以猜对，所以对于我个人来说有必要对其的使用进行实际练习和总结，对其产生清晰的认识。
  • 其外，目前网络上大部分博客资料都是针对iOS的，Mac虽然与其有一定相似性但是我个人认为还是有必要单独写一篇针对Mac的讲解，毕竟作为一个Mac开发初学者，经历过查阅资料只有针对iOS开发的窘境，而实际使用又需要自己去探索Cocoa是否支持，操作起来有什么差异等等。

二、Mac的坐标系统

OSX系统中视图坐标系统：

是不是和数学中常用的坐标轴是一致的，但是可能对于很多习惯iOS系统视图坐标的人来说，可能更习惯视图坐标原地那在左上角，可以通过重写视图的isFlipped函数设置：

- (BOOL)isFlipped{              //重写在自定义的NSView子类  
    return YES;
}

注：该方法需重写在自定义的NSView的子类中，同时它只能影响以该视图为父视图的视图（下面有例子）。

三、frame的使用

Cocoa中使用NSRect给frame或bounds赋值,NSRect其实就是一个C语言的结构：

typedef CGRect NSRect;          //NSRect其实是CGRect的别名
struct CGRect {
    CGPoint origin;             //原点坐标          
    CGSize size;                //宽高大小
};
struct CGPoint {
    CGFloat x;                  //x轴坐标
    CGFloat y;                  //y轴坐标
};
struct CGSize {
    CGFloat width;              //宽度
    CGFloat height;             //高度
};

视图的frame定义为视图在其父视图的坐标系的坐标和大小(其中坐标指的是该视图的左下角)：

NSView *blueView = [[NSView alloc] initWithFrame:NSMakeRect(30, 50, 250, 300)];
blueView.wantsLayer = YES;
[blueView.layer setBackgroundColor:[NSColor blueColor].CGColor];
/* 将blueView作为NSWindow的contentView的子视图 */
[[self.window contentView] addSubview:blueView];

NSView *redView = [[NSView alloc] initWithFrame:NSMakeRect(20, 20, 50, 100)];
redView.wantsLayer = YES;
[redView.layer setBackgroundColor:[NSColor redColor].CGColor];
/* 将redView作为blueView的子视图 */
[blueView addSubview:redView];

  • blueView是整个窗口的contnetView的子视图，所以其frame的坐标的指定原点为contnentView的原点；
  • 同理redView是blueView的子视图，其frame设置的是在blueView的坐标系中的位置；
  • 同时可以注意到，frame中size的大小是不受视图所在坐标系影响的(单位长度都是一样的)。

  • flipped视图的区别：

    其中代码实现的唯一改动是blueView是我自定义重写了isFlipped函数的NSView的子类。可以看见当设置blueView的frame时，contnetView的坐标系不变，所以blueView还是在原位置；而blueView视图坐标系统改变导致其下子视图位置改变。

  • tips：
    Q：redView的子视图会有影响吗？
    A：不会，由于redView的isFlipped返回值仍是NO，所以其子视图还是以其左下角为原点。

四、bounds的解释

可能到这里你会和我一样感到奇怪，貌似一个frame已经够用了：我们已经可以将视图放到我们想要的位置了，那为什么还需要bounds呢？

这里先引用一下苹果官方文档对bounds的解释：

The view’s bounds rectangle, which expresses its location and size in its own coordinate system.

如果用中文来翻译的话：bounds——表示其自身坐标系统中的位置和大小。这是一句耐人寻味的话，很难从定义中明白其存在意义，至少初学时我是不太理解的，所以还是需要代码试验和理解：

NSView *redView = [[NSView alloc] initWithFrame:NSMakeRect(100, 120, 50, 100)];
redView.wantsLayer = YES;
[redView.layer setBackgroundColor:[NSColor redColor].CGColor];
[[self.window contentView] addSubview:redView];
NSLog(@"%@", NSStringFromRect(redView.bounds));

输出：{{0, 0}, {50, 100}}

可以看出视图的bounds默认origin为(0, 0),size为frame的size(实际中bounds的getter函数就是这么实现的)。

接下来我们试验一下修改bounds会发生什么：

可以看到修改bounds对view本身对frame没有任何影响，那我们尝试在其内加入子视图：

  • 修改origin：

    修改父视图的bounds的origin之后，父视图的位置不变(父视图frame未变且父视图所在坐标系统未变)，而对于其子视图所在坐标系(即父视图坐标系)，虽然子视图frame未变，但是其所在坐标系原点改变，则相同frame下子视图和父视图相对位置发生变化，所以视觉上其显示位置改变。

  • 修改size:

    修改父视图的bounds的size后，其子视图大小有明显变换。在本例中，父视图的width变为原来的4/5，即200，而子视图的width为100即其一半，因而子视图的width应该占整个父视图的一半(height同理)，所以实际显示中子视图的size都成比例的增加了！(尽管子视图的frame并没有改变，但是其所在坐标系统的size变了)

  • bounds的作用
    经过上例的验证后我们可以发现，官方对于bounds的解释如此简练，光凭其定义想要去理解是有难度的。但如果从其作用的角度，我们可以将bounds定义为：决定视图本身的坐标系统，该坐标系统会影响其内视图的呈现，即影响视图的子视图。

五、实例：NSScrollView解析

官方提供的控件——滚动视图中应用了bounds的一定特性，在此进行解析，了解滚动视图的实现和bounds的应用。

  • NSScrollView控件的结构：

    上图可以看到一个滚动视图控件包含好几个部分，其中绿色部分(NSView)就是我们承载显示内容的部分，其size一般都比ContentView大，这样才有滚动显示的需求。

先将一个image添加到scrollView的documentView中，给视图添加一个按钮，点击按钮修改不同View的bounds查看效果：

  • 显示部分的代码：

    /* 该图片事先拖入了.xcassets中 */
NSImage *deerImage = [NSImage imageNamed:@"deer"];
/* scorllView的bounds默认(0,0,frame.width,frame.height) */
NSImageView *imageView = [[NSImageView alloc] initWithFrame:self.scrollView.bounds];
imageView.image = deerImage;
[imageView setFrameSize:deerImage.size];
self.scrollView.documentView = imageView;

/**
*  前面设置imageView的frame时，origin为(0, 0)，显示时会先显示左下角内容，
*  不符合习惯，所以计算其左上角显示的点，利用scrollView的api滚动到对应位置。
*/
NSPoint topleftPoint = NSMakePoint(0., imageView.frame.size.height - self.scrollView.contentView.frame.size.height);
[self.scrollView.contentView scrollToPoint:topleftPoint];

  • 下面是修改不同bounds内容对应的效果：

    • 修改scrollView的bounds

      - (IBAction)onButtonClicked:(id)sender {
   NSRect rect = self.scrollView.bounds;
   rect.origin.x += 50;
   self.scrollView.bounds = rect;
}

      实际整个contentView(doucumentView + 两个滚动条)都移动了，而contnetView内部视图相对布局没有改变。

      - (IBAction)onButtonClicked:(id)sender {
   NSRect rect = self.scrollView.bounds;
   rect.size.width *= 2;
   rect.size.height *= 2;
   self.scrollView.bounds = rect;
}

      整个contentView缩小了。(其父视图坐标系统bounds的size加大)

    • 修改其contentView的bounds

      - (IBAction)onButtonClicked:(id)sender {
   NSRect rect = self.scrollView.contentView.bounds;
   rect.origin.x += 50;
   self.scrollView.contentView.bounds = rect;
}

      修改contnetView的bouds的origin，图片移动，其效果等于拖动滚动条。

      - (IBAction)onButtonClicked:(id)sender {
   NSRect rect = self.scrollView.contentView.bounds;
   rect.size.width *= 2;
   rect.size.height *= 2;
   self.scrollView.contentView.bounds = rect;
}

      修改contnetView的bouds的size，图片缩小/放大，其效果等于给scrollView加上一个放大缩小按钮。

    • 修改其documentView的bounds

      - (IBAction)onButtonClicked:(id)sender {
   NSRect rect = self.scrollView.documentView.bounds;
   rect.origin.x += 50;
   rect.size.width *= 2;
   self.scrollView.documentView.bounds = rect;
}


      修改documentView的bounds，可以看见图片似乎和修改contnetView的bounds效果一样，但是注意到滚动条位置，可以发现我们之前总结的没有问题，修改doucuemntView的bounds后，根其同级的滚动条是没有影响的，只会影响documentView内呈现内容的视图(就是它的子视图)的位置！

六、总结

通篇我们只能总结两句话：

  • frame决定视图在其父视图中的位置和大小。
  • bounds决定视图本身的坐标系统，影响其子视图的位置和大小。(iOS中bounds和frame的size是保持一致的，很不一样)

而在实际探究中我发现当我们学习某一常用概念时，很可能会想当然地接受，导致没有深究，而忽略了一些细节。
